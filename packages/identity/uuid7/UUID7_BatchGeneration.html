High-Efficiency Batch UUIDv7 Generation

📚 Navigation: Home > Dixor.Identity > Batch Generation

When inserting thousands of database records, seeding test environments, or running high-throughput ETL pipelines, generating primary keys one by one inside a standard for loop introduces unnecessary boilerplate and memory allocation overhead.

Dixor.Identity streamlines bulk identifier creation via memory-efficient, lazily evaluated batch generation:

IEnumerable<Guid> batchIds = Uuid7.NewGuids(count: 10000);

Why Lazy Evaluation Matters

Rather than instantly allocating a massive array in heap memory (Eager Generation), Uuid7.NewGuids() utilizes C# iterator state machines (yield return) to stream identifiers on demand.

EAGER GENERATION (Memory Heavy):
[Call API] ──► Allocates 1,000,000 GUIDs in RAM instantly ──► [Process Array]

LAZY STREAMING (Memory Efficient - Dixor Approach):
[Call API] ──► Yields GUID #1 ──► [Process #1] ──► Discards #1 ──► Yields GUID #2 ...

Key Architectural Advantages

  1. Zero Heap Bloat: Requesting 10_000_000 identifiers consumes virtually zero memory up front. Identifiers materialize strictly as your consumer (foreach, LINQ, or database bulk writer) requests them.
  2. Reduced GC Pressure: Eliminating large temporary arrays prevents Garbage Collector spikes and memory fragmentation during intense data migrations.
  3. Pipeline Composable: Works seamlessly with standard LINQ streaming queries (Where, Take, Chunk).

API Usage & Examples

Basic Streaming

using Dixor.Identity.UUID7;

// Stream 5,000 time-ordered keys directly into a database bulk inserter
foreach (Guid primaryKey in Uuid7.NewGuids(5000))
{
    await dbContext.Orders.AddAsync(new Order { Id = primaryKey });
}

Materializing When Necessary

If your downstream API explicitly demands an in-memory collection (such as serialization payloads), materialize the stream explicitly:

List<Guid> orderIds = Uuid7.NewGuids(count: 500).ToList();
Guid[] idArray      = Uuid7.NewGuids(count: 100).ToArray();

Compile-Time Safety: Roslyn Analyzer (DIXOR001)

To prevent runtime application crashes during bulk operations, Dixor.Identity includes a dedicated compile-time Roslyn analyzer rule (DIXOR001) that inspects batch sizes directly inside your IDE.

Diagnostic Rule Description Trigger Condition Resolution
DIXOR001 Invalid batch generation count Passing count <= 0 as a compile-time constant Supply a positive integer (> 0) 
// ❌ COMPILE ERROR (DIXOR001): Batch size must be greater than zero
var badBatch = Uuid7.NewGuids(0);

// ✅ COMPILE SUCCESS: Valid positive batch count
var goodBatch = Uuid7.NewGuids(500);

Best Practices

  • Prefer Streaming Over Materialization: Avoid calling .ToList() or .ToArray() on large batches unless your destination API strictly demands a materialized collection.
  • Validate Dynamic Inputs: While Roslyn catches invalid compile-time constants, always validate batch sizes originating from HTTP payloads or external user input before invoking generation:
if (requestCount <= 0)
    return Results.BadRequest("Batch count must be a positive integer.");
