UUIDv7 Production Best Practices

📚 Navigation: Home > Dixor.Identity > Best Practices

Adopting UUID Version 7 immediately supercharges database indexing and chronological observability compared to random GUIDs. However, getting the most out of Dixor.Identity in high-throughput production environments requires following established architectural patterns.

This guide outlines essential best practices for database storage, API boundary validation, and high-concurrency generation.

1. Database Storage & Schema Design

Always Store as Native 128-Bit Binary Types

Never store UUIDs as text strings (VARCHAR(36) or NVARCHAR(36)) in relational databases. String storage doubles disk space consumption, ruins CPU cache locality, and slows down index traversals.

Database Engine Recommended Column Type Why?
SQL Server UNIQUEIDENTIFIER 16-byte native storage; optimized for B-Tree sequential leaf insertion.
PostgreSQL UUID First-class binary UUID handling with native operator support.
MySQL / MariaDB BINARY(16) Fixed-length raw binary layout ensuring tight index page packing. 
-- Optimal SQL Server Schema Example
CREATE TABLE Orders
(
    Id UNIQUEIDENTIFIER PRIMARY KEY NOT NULL,
    CustomerId UNIQUEIDENTIFIER NOT NULL,
    OrderTotal DECIMAL(18, 2) NOT NULL
);

2. Choosing the Right Generation Strategy

Match your generation method to the exact concurrency profile of your workload:

using Dixor.Identity.UUID7;

// 1. STANDARD WORKLOADS (REST APIs, Entity Creation)
// Ultra-fast, zero-allocation time-ordered ID.
Guid entityId = Uuid7.New();

// 2. ULTRA-HIGH THROUGHPUT (Event Sourcing, High-Speed CQRS)
// Guarantees strict mathematical sorting even when 10,000+ IDs are minted in the exact same millisecond.
Guid eventId = Uuid7.NewMonotonic();

// 3. BULK OPERATIONS (Data Seeding, ETL Pipelines, Batch Inserts)
// Lazy enumeration eliminates loop overhead and memory pressure.
IEnumerable<Guid> seedIds = Uuid7.NewGuids(count: 5000);

3. API Boundary Safety & Input Validation

Never Rely on Exception-Driven Parsing

When receiving identifiers from external REST payloads, query parameters, or file uploads, never treat input as trusted data. Using Uuid7.Parse() directly on user input risks throwing unhandled exceptions under malformed payloads.

// ❌ BAD: Exception-driven flow incurs severe CPU overhead on bad requests
try 
{
    Guid id = Uuid7.Parse(rawInput);
} 
catch (FormatException) { /* Handle error */ }

// ✅ GOOD: Fast, allocation-free validation pattern
if (!Uuid7.TryParse(rawInput, out Guid validId) || !Uuid7.IsValid(validId))
{
    return Results.BadRequest("Invalid RFC 9562 UUIDv7 identifier provided.");
}

4. Architectural Patterns & Domain Formatting

Decouple Generation via Dependency Injection

To keep domain logic clean and unit tests deterministic, wrap static generation calls behind a lightweight provider interface:

public interface IIdGenerator
{
    Guid CreateId();
}

public sealed class DixorIdGenerator : IIdGenerator
{
    public Guid CreateId() => Uuid7.New();
}

// Register in ASP.NET Core Program.cs:
builder.Services.AddSingleton<IIdGenerator, DixorIdGenerator>();

Use Built-in Domain Formatting APIs

Avoid manual string concatenation when generating business-facing reference codes. Let the allocation-conscious formatter handle prefixing and separators:

// ❌ BAD: Allocates multiple temporary strings
string badRef = "ORD-" + Uuid7.New().ToString() + "-US";

// ✅ GOOD: Optimized internal string formatting
string cleanRef = Uuid7.NewString(prefix: "ORD", suffix: "US");
// Output: ORD-0197d8ef-f4c6-7a35-99e3-1d4a7e3c8e10-US

5. Security & Temporal Considerations

  • Do Not Rely on UUIDs for Authorization: Because UUIDv7 embeds a public creation timestamp, observers can deduce roughly when a record was created. Never use UUIDs as secret bearer tokens or security nonces.
  • Preserve UTC Strictly: Extracted timestamps (Uuid7.GetTimestamp(id)) return DateTimeOffset anchored in UTC. Always preserve UTC across distributed service boundaries and convert to local time strictly at the UI layer.
  • Not a Sequential Counter Across Nodes: While UUIDv7 sorts chronologically, two independent distributed microservices minting IDs during the exact same millisecond rely on random lower bits. Never assume Node_A_ID + 1 == Node_B_ID.

Production Quick-Check Matrix

Common Anti-Pattern Production Best Practice
Storing UUIDs as VARCHAR(36) Use native 16-byte binary columns (UNIQUEIDENTIFIER / UUID).
Looping Uuid7.New() 10,000 times Use Uuid7.NewGuids(10000) for memory-efficient lazy streaming.
Using Parse() on HTTP payloads Use TryParse() + IsValid() to prevent exception overhead.
Replacing hyphens (id.ToString().Replace("-", "")) Keep standard canonical formatting or use Uuid7.NewString().
Assuming sub-millisecond sorting across threads Switch to Uuid7.NewMonotonic() for strict sequence locking.
