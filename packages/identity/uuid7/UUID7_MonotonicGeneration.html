Monotonic UUIDv7 Generation (Uuid7.NewMonotonicGuid)

📚 Navigation: Home > Dixor.Identity > Monotonic Generation

While standard Uuid7.New() utilizes random bits for lower payloads, high-throughput applications generating thousands of identifiers within the same millisecond can experience out-of-order sorting. Dixor.Identity solves this via Monotonic Generation:

using Dixor.Identity.UUID7;

// Guarantees strict sequential ordering across same-millisecond invocations
Guid eventId1 = Uuid7.NewMonotonicGuid();
Guid eventId2 = Uuid7.NewMonotonicGuid(); // Guaranteed > eventId1

How Monotonic Sequencing Works

The underlying Uuid7MonotonicGenerator uses an exclusive lock (_lock) and an internal 16-bit sequence counter (_counter) to manage same-millisecond bursts.

The Same-Millisecond Collision Problem

Consider an application inserting records inside a tight loop:

// Standard Uuid7.New() inside the same millisecond (e.g., 1751132765000 ms)
0197d8ef-f4c6-7a35-... (Random lower bits)
0197d8ef-f4c6-7f91-... (Random lower bits)
0197d8ef-f4c6-7210-... (Random lower bits)

Because the lower bits are filled with cryptographic entropy, database index sorting may arrange them out of insertion order (...7210, then ...7a35, then ...7f91).

How Monotonic Sequencing Works

To preserve mathematical ordering across identical timestamps, Uuid7.NewMonotonicGuid() utilizes thread-safe synchronization to manage an internal 16-bit sequence counter (ushort _counter):

Same Millisecond (1751132765000 ms):
[Call #1] ──► Timestamp: 1751132765000 | Counter: 0
[Call #2] ──► Timestamp: 1751132765000 | Counter: 1
[Call #3] ──► Timestamp: 1751132765000 | Counter: 2

Clock Advances to New Millisecond (1751132765001 ms):
[Call #4] ──► Timestamp: 1751132765001 | Counter: 0 (Reset)

Bitwise Layout & Counter Embedding

  • Bytes 14 & 15: Dedicated to the 16-bit sequence counter in Network Byte Order, accommodating up to 65,536 strictly ordered identifiers per millisecond.
  • Global Uniqueness: Bytes 6 through 13 continue to receive high-entropy randomness via RandomNumberGenerator.Fill().

  • Throughput Capacity: A 16-bit counter accommodates up to 65,536 strictly ordered identifiers per millisecond (over 65 million IDs per second per process).
  • Cryptographic Uniqueness: Bytes 6 through 13 continue to receive high-entropy randomness via RandomNumberGenerator.Fill(), preserving global collision resistance across distributed nodes.

Standard vs. Monotonic Generation

Feature Standard (Uuid7.NewGuid()) Monotonic (Uuid7.NewMonotonicGuid())
Same-Millisecond Ordering Random Strictly Sequential
Execution Overhead Zero Lock / Sub-Nanosecond Minimal Lock Contention
Best Used For REST APIs, User Creation Event Sourcing, High-Speed CQRS
