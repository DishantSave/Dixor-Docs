ULID Parsing & Binary Conversion (Ulid.Parse / Ulid.TryParse)

📚 Navigation: Home > Dixor.Identity > Parsing

Dixor.Identity provides high-performance, allocation-free mechanisms for parsing canonical 26-character Universally Unique Lexicographically Sortable Identifier (ULID) strings back into their original 16-byte binary representations via Ulid.Parse and Ulid.TryParse.

using Dixor.Identity.ULID;

// Strict parsing (throws FormatException on malformed input)
byte[] bytes = Ulid.Parse("01J5J8Y9Q6A8K5J8X3Z5N9Y2KP");

// Safe parsing without exceptions
if (Ulid.TryParse("01J5J8Y9Q6A8K5J8X3Z5N9Y2KP", out byte[]? parsedBytes))
{
    // Successfully decoded into 16 bytes
}

Architectural Mechanics & Fixed-Index Bit Unmapping

The parsing engine acts as the inverse of the formatting pipeline, transforming Crockford Base32 strings back into a 128-bit structure:

26 Crockford Base32 Characters
    ↓
Fixed-Index Bit Unmapping
    ↓
128-bit Binary Representation (16 bytes)
  1. Fixed-Index Unmapping: Uses exact inverse bit-shifting corresponding to the encoder layout to reconstruct the 16 bytes precisely without offset drift.
  2. Human-Error Tolerance (Crockford Aliases): In addition to standard Base32 characters, the parser natively accepts visually ambiguous characters (I, L, and O) as aliases for 1 and 0, improving readability and transcription tolerance.

Validation Rules Enforced During Parsing

Before converting the string, the parser executes strict validation checks:

  • Length Constraints: The input string must contain exactly 26 characters.
  • Overflow Limits: The first character is checked to ensure its decoded numeric value does not exceed 7, preventing high-bit range overflows beyond the 128-bit boundary.
  • Alphabet Membership: Every character is verified against the Crockford Base32 alphabet. Any unsupported characters trigger a FormatException.
