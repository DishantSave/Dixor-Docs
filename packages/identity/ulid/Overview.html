ULID (Universally Unique Lexicographically Sortable Identifier)

Introduction

The Dixor.Identity.ULID namespace provides high-performance, specification-compliant generation, parsing, validation, formatting, and timestamp extraction for 128-bit Universally Unique Lexicographically Sortable Identifiers (ULIDs).

ULIDs combine the global uniqueness of UUIDs with the chronological sorting capabilities required by modern distributed databases, event logs, and high-throughput microservices.

Architecture & Layout

A ULID consists of exactly 128 bits (16 bytes) divided into two core components:


+------------------------------------+------------------------------------+
| Timestamp Component                | Randomness Component               |
| 48 bits (6 bytes)                  | 80 bits (10 bytes)                 |
| Big-endian Unix time in millis     | Cryptographically secure random    |
+------------------------------------+------------------------------------+
  1. 48-Bit Timestamp (Bytes 0–5): Encodes the Unix epoch timestamp in milliseconds. Because the timestamp occupies the most significant bytes, IDs generated sequentially are naturally sortable in chronological order.
  2. 80-Bit Randomness (Bytes 6–15): Generated via cryptographic random number generators (RandomNumberGenerator), ensuring collision resistance within and across millisecond bounds.
  3. Crockford Base32 Encoding: The complete 128-bit structure is encoded into a concise 26-character canonical string using the Crockford Base32 alphabet.

Core Capabilities

  • Standard Generation: Create timestamp-backed identifiers instantly using system UTC time or explicit timestamp overrides.
  • Monotonic Sequencing: Guarantee strictly increasing order even when multiple identifiers are generated within the exact same millisecond window.
  • High-Efficiency Batching: Lazily evaluate and yield multiple identifiers for high-throughput bulk operations.
  • Deterministic Bit Mapping: Utilize fixed-index bit mapping to ensure complete compatibility with external specification-compliant libraries without offset drift.
  • Allocation-Free Parsing & Formatting: Leverage Span<T> and stack allocation (stackalloc) to reduce heap pressure.
